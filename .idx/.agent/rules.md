# Project Rules and Best Practices

This document outlines the rules and best practices for contributing to this project. All contributors, including the AI assistant, are expected to adhere to these guidelines.

## The Golden Rule

**Every time an edit is made, it must be committed and pushed to GitHub.** This ensures that all changes are tracked and that the codebase can be easily reverted to a previous state if necessary.

## Additional Rules

*   **Clear Commit Messages:** All commit messages should be clear, concise, and follow the [Conventional Commits](https://www.conventionalcommits.org/en/v1.0.0/) specification. This helps to maintain a clean and readable commit history.
*   **Branching Strategy:** All new features should be developed on a separate feature branch. Once the feature is complete, it should be merged into the `main` branch via a pull request. This allows for code review and ensures that the `main` branch always contains stable code.
*   **Code Style:** All code should adhere to a consistent code style. This makes the code easier to read and maintain. We use Prettier for automatic code formatting.
*   **Testing:** All new code should be accompanied by unit tests. This helps to ensure that the code is working correctly and prevents regressions.
*   **AI Assistant Integration:** The AI assistant is a valuable member of the team, but all code generated by the assistant should be reviewed by a human developer before it is merged into the `main` branch.

# Development Rules
- **TypeScript:** Use strict typing. Never use `any`.
- **Database:** All database changes must be reflected in `prisma/schema.prisma` and configured via `prisma.config.ts`.
- **Secrets:** Never log the contents of `google-key.json` or show it in the UI.
- **Performance:** Ensure all images are optimized using the Next.js `<Image />` component.
